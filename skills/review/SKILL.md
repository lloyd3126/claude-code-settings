# 代碼審查工作流

## 審查步驟

1. **取得變更範圍**：執行 `git diff` 或 `git diff --staged` 查看所有變更
2. **逐文件審查**，檢查以下維度：

### 正確性
- 邏輯是否正確？有無明顯 bug？
- 邊界條件是否處理？
- 錯誤處理是否完善？

### 安全性
- 是否有注入風險（SQL、XSS）？
- 敏感資料是否暴露？
- 認證/授權是否正確？

### 效能
- 是否有 N+1 查詢？
- 是否有不必要的重渲染？
- 大數據量處理是否合理？

### 可維護性
- 命名是否清晰？
- 是否遵循專案的命名規範？
- 是否有重複代碼可抽取？

### 測試
- 新功能是否有測試覆蓋？
- 測試是否覆蓋了邊界情況？

3. **輸出審查報告**，格式：

```
## 審查摘要
- 變更文件數：X
- 總體評價：✅ 通過 / ⚠️ 需修改 / ❌ 需重做

## 問題清單
| # | 文件 | 行號 | 嚴重程度 | 描述 | 建議 |
|---|------|------|---------|------|------|

## 優點
- （列出做得好的地方）
```
