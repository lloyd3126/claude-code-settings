# 初始化專案 Steering 文件

請依序執行以下步驟：

## 第一步：掃描專案

掃描當前專案的以下內容：
- 目錄結構
- `package.json`（或同等依賴管理文件）
- 配置文件（tsconfig.json、.eslintrc、prisma/schema.prisma 等）
- 已有的代碼和功能模組
- `.env.example`（如存在）

## 第二步：建立目錄

```bash
mkdir -p doc/steering
mkdir -p doc/specs
```

## 第三步：生成四個 Steering 文件

根據掃描結果（非模板示例值）生成：

### 1. `doc/steering/product.md` — 產品概覽
- 產品名稱與使命
- 目標用戶
- 核心功能列表
- 當前各模組完成狀態（✅ / 🔧 / ⬜）
- 業務目標
- 非功能性需求（性能、可用性、多語言等）

### 2. `doc/steering/tech.md` — 技術棧說明
- 整體架構描述（各層如何協作）
- 前端/後端/資料庫/基礎設施的具體技術和版本
- 關鍵架構決策紀錄（選了什麼、為什麼選）
- 開發工具和注意事項

### 3. `doc/steering/structure.md` — 目錄結構
- 實際目錄樹（基於掃描結果）
- 各目錄的職責邊界（什麼代碼放哪裡）
- 命名規範（文件、變數、常數、類型）
- Import 順序規範

### 4. `doc/steering/progress.md` — 專案進度總覽與技術債
此文件是「鳥瞰圖 + 索引」，不追蹤任務細節（那是 Spec 的 tasks.md 負責的）。
- **功能狀態一覽表**：列出所有功能、狀態（✅/🔧/⬜）、對應的 Spec 路徑
- **技術債 / 已知問題**：不屬於任何功能 Spec 的問題
- **非 Spec 的修復與改進紀錄**：小修復、hotfix、不走 Spec 流程的變更
- **里程碑**：版本發布計劃、重要截止日期

## 第四步：建立專案級 CLAUDE.md

在專案根目錄建立 `CLAUDE.md`（如不存在），內容精簡（< 100 行）：
- 專案名稱和一句話描述
- 當前狀態速覽
- 技術棧速覽（框架 + 語言 + 工具）
- 架構一句話總結
- 核心開發規則（包管理器、測試命令等）
- 指向 doc/steering/ 各文件的 @path 引用
